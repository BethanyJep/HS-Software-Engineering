# Session 1 – Introduction to Software Engineering

**Term:** 2  
**Duration:** 90 minutes  
**Audience:** Grade 10 students, Alliance Girls High School  
**Facilitators:** TOFA / Alumni volunteers (2–3 recommended)

---

## Session Objectives

By the end of this session, students will be able to:

1. Define software engineering in their own words.
2. Describe the basic stages of building software (the software development lifecycle).
3. Name at least three roles that exist on a software team.
4. Feel curious and excited about exploring software engineering further.

---

## Materials Needed

- [ ] Projector / display screen
- [ ] Facilitator laptop (fully charged, demos tested beforehand)
- [ ] Printed "Software Roles" cards (one set per table group) — see Appendix A
- [ ] Sticky notes (two colours) + pens
- [ ] Whiteboard or flip chart
- [ ] Student handout: "What I Learned Today" exit ticket — see Appendix B
- [ ] Optional: small sweets/stickers as prizes for activities

## Session Plan

### ⏱ 0–10 min | Welcome & Icebreaker

**Facilitator action:** Welcome students, introduce TOFA/Alumni, and run an icebreaker.

> 💡 See [Icebreakers](../resources/icebreakers.md) for options. **Recommended for this session:** *Two Truths and a Tech Lie* or *Human Bingo*.

Transition line:
> *"Today we're going to talk about one of the most exciting and in-demand careers in the world – software engineering. And by the end of this session, you'll understand what it actually is – because it's not just 'coding'."*

---

### ⏱ 10–25 min | What Is Software Engineering?

**Format:** Interactive mini-lecture (facilitator talks, students respond)

**Key questions to pose to the room:**
- "What apps or websites did you use in the last 24 hours?" *(Get hands up, write answers on the board)*
- "Who do you think made those? How?"

**Core concepts to cover:**

1. **Software vs. Hardware**
   - Hardware = physical (phone, laptop, keyboard)
   - Software = programmes, apps, systems that run on hardware
   - Software engineering = the discipline of *designing, building, and maintaining* software

2. **Software is everywhere**
   - Banking apps, hospital systems, weather forecasts, social media, exam marking systems, navigation
   - "Software is eating the world" – Marc Andreessen

3. **Software Engineering ≠ just coding**
   - Analogy: Building a house needs architects, engineers, electricians, plumbers, interior designers — not just one person with a hammer.
   - Software needs designers, developers, testers, project managers, data analysts, security experts.

4. **The Software Development Lifecycle (SDLC)** – draw this on the board as you explain:
   ```
   Plan → Design → Build → Test → Deploy → Maintain
   ```
   Use a relatable example as you walk through each stage:
   > *"Imagine you're building an app to help students at this school swap books. Let's walk through each stage…"*

---

### ⏱ 25–40 min | 🔥 Live Demo – Seeing Software in Action

**Purpose:** Make software tangible and exciting. Show something visual that students can react to.

> 💡 See [Cool Demos](../resources/demos.md) for full instructions. **Recommended for this session:**
> - **Demo 2:** Add a button that responds to a click
> - **Demo 3:** Update page colours with a selector
> - **Demo 4:** Show and hide a message with a toggle

**Facilitation tips:**
- Narrate as you code/demo: *"I'm typing instructions the computer will follow exactly."*
- Make a deliberate mistake and let students spot/fix it — this normalises debugging.
- Ask: *"What would you change about this? What would YOU build?"*

---

### ⏱ 40–55 min | Software Roles Activity

**Format:** Small group activity (groups of 4–5)

**Instructions:**
1. Give each group a set of **Software Roles Cards** (see Appendix A).
2. Each card names a role (e.g., Front-End Developer, UX Designer, Data Analyst, DevOps Engineer, Product Manager, Security Engineer, QA Tester).
3. Groups read the cards and discuss:
   - *"Which of these roles sounds most interesting to you, and why?"*
   - *"Which roles do you think need you to write code? Which ones don't?"*
4. Each group shares one insight with the room.

**Facilitator debrief points:**
- Many roles in tech don't require deep coding — there is a path for everyone.
- Skills like communication, empathy, and problem-solving are just as important as technical skills.
- The roles work together — no one builds software alone.

---

### ⏱ 55–70 min | Q&A + Discussion

**Prompt students with:**
- *"What surprised you most today?"*
- *"Is there anything about software engineering that sounds hard or scary?"*
- *"Can you think of a problem at your school or in your community that software could solve?"*

Write the last question's answers on the board — these become seeds for the Term 3 project!

---

### ⏱ 70–80 min | Wrap-Up & Next Steps

**Key messages to reinforce:**
- Software engineering is a creative and collaborative discipline.
- You don't need to be a genius or a "math person" — curiosity and persistence matter most.
- In the next session, you'll hear from real people who work in tech.

**Announce Session 2:**
> *"Next time, we're bringing in some people who actually work in tech — including some Alliance Girls alumni — to tell you their story. Come with questions!"*

---

### ⏱ 80–90 min | Exit Ticket

Hand out the **"What I Learned Today"** slip (Appendix B).

Students fill in:
1. One thing I learned today: ___
2. One question I still have: ___
3. On a scale of 1–5, how excited am I about software engineering? ⭐

Collect slips — use the questions to shape the next session's Q&A.

---

## Appendix A – Software Roles Cards

*Print and cut. One set per group of 4–5 students.*

| Role | What they do |
|---|---|
| **Front-End Developer** | Builds what users see and interact with (buttons, screens, animations) |
| **Back-End Developer** | Builds the "engine" behind the scenes — databases, logic, APIs |
| **Full-Stack Developer** | Does both front-end and back-end |
| **UX/UI Designer** | Designs how the app looks and feels; makes it easy to use |
| **Product Manager** | Decides what the team builds and why; represents the user |
| **QA / Test Engineer** | Tests the software to find and document bugs |
| **Data Analyst / Engineer** | Works with data — finding patterns, building dashboards |
| **DevOps / Cloud Engineer** | Makes sure the software runs reliably on servers |
| **Security Engineer** | Protects the software from hackers and vulnerabilities |
| **Technical Writer** | Writes documentation so others can understand and use the software |

## Facilitator Notes

- Keep energy high — this is students' first impression of the program.
- Don't worry if you can't answer every question; model intellectual curiosity: *"Great question — let's find out together."*
- Watch for students who seem disengaged or intimidated; pair them with an enthusiastic peer.
- If the demo breaks, that's actually a great teaching moment about debugging!
- Collect exit tickets and share key themes with co-facilitators before Session 2.
